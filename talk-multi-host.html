drishti — multi-host tabs

Multi-host viewing with a tab strip. Reviewed against Lowy (volatility) and Hickey (structural simplicity) before this artifact.

1. Where the constraint actually lives

The session/transport layer is already multi-host: getHostSession({host, drvPath, binary}) is pool-keyed by that triple, and identical callers share one ssh subprocess. /tmp/kolu-drishti/packages/surface-nix-host/src/hostSession.ts:371,385

The single-host limitation lives entirely in drishti:

  • packages/app/src/server/main.ts:36 reads exactly one HOST from argv/env, builds one session and one router, mounts one WS.
  • packages/app/src/server/router.ts — buildRouter({session}) is closure-scoped; instantiating it N times gives N independent caches/buses with zero shared state between hosts.
  • packages/app/src/client/wire.ts exports a singleton app bound to one PartySocket.
  • packages/app/src/client/App.tsx reads from that singleton at the top level.

2. Proposed shape

Keep the per-host surface schema unchanged. Multiplex hosts at the transport layer: one HostSession + one router fragment + one client-side surfaceClient per host. Host identity lives in the WS URL, never inside the surface keys.

3. Reviewer pass — what changed in the sketch

Both reviewers ran before this artifact. Their substantive bites are folded in below; the headline design above already reflects them.

Adopted — drop processCount on SystemInfo. REJECTED

Earlier sketch added a processCount: number field to SystemInfo so inactive tabs could render a count badge without holding the heavy processesSnapshot stream open. Both reviewers landed on this as the worst structural risk: Hickey called it concept multiplication forcing either two writers to one cell (parent mutates agent-owned data) or fragmentation by construction (count and rows stay in sync across two surface entities). Lowy called it a god-cell-in-the-making driven by UI display volatility leaking into per-host data shape. Decision: Phase 1 ships connection-state dot only on inactive tabs — no count badge until the user actively switches to that tab. If process count on inactive tabs becomes a real ask, derive it client-side from the existing processesSnapshot traffic; don't grow the surface schema.

Adopted — kill /hosts.json, use a tiny admin surface instead. FOLDED IN

Lowy: a one-off REST endpoint for host discovery is a third protocol alongside WS+surface, and Phase 2's admin surface would replace it anyway. Ship the admin surface in Phase 1 with a read-only hosts collection. Phase 2 just adds the mutation procedures. Same protocol throughout.

Adopted — query-param dispatch, not path-segment. FOLDED IN

Earlier sketch routed by URL path (/rpc/ws/<host>). Lowy: that bakes host identity into the server's routing table, making the table volatile with Phase 2's dynamic add/remove. Switched to /rpc/ws?host=<id> — single endpoint, server picks the per-host RPCHandler off the query.

Adopted — tab status is a function of connection.state, not a new enum. FOLDED IN

Hickey: inventing a TabStatus enum parallel to the existing ConnectionState would be a sum-type-as-parallel-fields that has to stay in lockstep. Dot color is computed directly from connection.state. One source of truth.

Adopted — surfaceForHost(host) factory seam in wire.ts. FOLDED IN

Lowy: without that seam, "render tab N" forks App.tsx instead of being a prop change. Replace the singleton app export with a factory so HostView is parametric on host.

Considered and not adopted — multiplex N hosts over one WebSocket. VERIFIED

Hickey flagged this as worth verifying before committing to N sockets. I grepped: createCellsClient({websocket}) in /tmp/kolu-drishti/packages/surface/src/client.ts:77-87 wraps exactly one socket via RPCLink. The library has no namespacing/topic primitive — "namespace" in the source refers only to the internal surface.<key>.<verb> oRPC prefix, not multi-tenant multiplexing. N sockets is the framework-aligned shape, not reinvention.

Rejected outright — Option A (host-keyed surface). REJECTED

The alternative I sketched was: change the surface itself so every key has a host dimension (Collection<HostId, SystemInfo>, Collection<(HostId, Pid), Process>, etc), keep one WS. Both lenses rejected it: Lowy — complects two orthogonal volatility axes (host membership vs per-host data shape) into one schema; Hickey — forces host identity into every primitive's key, even though only the transport actually cares about it. Option B (above) keeps the two axes orthogonal.

4. Phase split

This is user-visible feature work that touches 5+ files. Phased so each phase is independently useful when merged.

Phase 1 — static multi-host with tabs ships alone

  • CLI: drishti host1 host2 host3. Single-host case (drishti alone, or one arg) is unchanged.
  • Server: N HostSessions, N router fragments, query-param-dispatched at /rpc/ws?host=<id>.
  • Admin surface (read-only) at ?host=__admin__ exposes the configured list — server reads its argv, exposes it as a hosts collection.
  • Client: surfaceForHost factory, MultiHostApp root, TabStrip, HostView parametric on host. Tab status = connection-state dot.
  • Subscription discipline: each TabChip holds a lifetime connection-cell subscription (very light). Active tab additionally holds system + processesSnapshot + cpuCores. Tab switch tears down the previous tab's heavy subs via onCleanup; the parent's caches stay warm because the bridge pumps are pinned regardless of browser subscriptions.
  • Shutdown: existing destroyAllSessions() already iterates the pool — no change needed.

What Phase 1 alone gives you: drishti web-01 db-02 edge-eu, open http://localhost:7720, tab strip shows all three with live connection-state dots, click any tab to see its full htop view. The full multi-host story minus the in-UI add/remove.

Phase 2 — dynamic add/remove from the UI deferred

  • Admin surface gains addHost(host: string) and removeHost(host: string) procedures.
  • Tab strip gains the "+" affordance (input field → addHost) and per-tab "×" (→ removeHost).
  • Host list persists: small JSON file in $XDG_STATE_HOME/drishti/hosts.json (or env-overridable). Loaded at server start, written on every add/remove.
  • Library nit: the kolu pool only exposes destroyAllSessions() publicly today. Phase 2 needs either a per-key destroyHostSession({host, drvPath, binary}) upstream, or drishti keeps its own map alongside the pool and calls session.destroy() directly. Trivial.

Cost of deferring Phase 2: you can't add a new host without restarting the parent process. For development this is negligible (drishti is already a just dev away from running with a new arg list). For long-running daemonized use it matters more.

5. Open question

Phase 1 deliberately ships no process count or other live metric on inactive tabs — only the connection dot. If "status on the tabs" means more than that to you (e.g. CPU% or process count visible at a glance across all tabs), the cleanest answer is to fold inactive-tab heavy subscriptions into Phase 1 instead of deferring; the agent→parent traffic is unchanged and the additional browser-bound delta traffic for ~5 hosts × 600 PIDs is on the order of 10 kB/s. Worth a sentence of feedback before I plan further.

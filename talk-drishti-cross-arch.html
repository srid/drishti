drishti — macOS → Linux: Exec format error

Why the darwin bun lands on the linux remote, and the shape of the fix after lowy + hickey review.

[host:pureintent remote] /nix/store/zwyyrxwlc05hq6j8qydv41p2qcc6svrq-drishti-agent/bin/drishti-agent: line 2:
/nix/store/421j82pv4gjmh9n2gchb4cs40hqfj20d-bun-1.3.10/bin/bun: cannot execute binary file: Exec format error

Root cause

The monitor wrapper bakes the local system's agent .drv path into one env var. Running on macOS, that's the aarch64-darwin derivation. When you add a Linux host, kolu ships that .drv across with nix copy --derivation, the Linux daemon substitutes the prebuilt darwin closure from cache.nixos.org, and the resulting wrapper script tries to exec a Mach-O bun on Linux.

The two-line trail

wherewhat
default.nix:46 makeWrapper ${resolvedPkgs.bun}/bin/bun $out/bin/drishti-agent — bun is captured from the evaluating nixpkgs.
default.nix:70 --set-default DRISHTI_AGENT_DRV "${drishti-agent.drvPath}" — scalar; encodes "one arch for all hosts" in the type.

The kolu library is explicit about this being the caller's job:

// nixCopy.ts:7-13
// No pre-built linux closure smuggled onto a darwin host.
// `nix eval --raw .#packages.<system>.<agent>.drvPath` is the typical recipe
// (probe the remote arch via `ssh $host uname -ms` first so the
// derivation is for the *remote's* architecture).

/nix/store/13al1sgi1bh7ygh970r6x8p9f98kq7ly-kolu-surface-nix-host/src/nixCopy.ts:7-13

Kolu deliberately takes a single drvPath in HostSessionOptions (hostSession.ts:52-64); arch selection is upstream of the library.

Deploy flow today

macOS monitor linux remote ───────────── ──────────── DRISHTI_AGENT_DRV = /nix/store/…-drishti-agent.drv (system = aarch64-darwin) │ ▼ getHostSession({ host, drvPath, binary }) │ ▼ provisionAgent ─── nix copy --derivation ──▶ daemon │ ▼ ssh host nix-store --realise │ ▼ substitute darwin closure from cache │ ▼ wrapper points at darwin bun │ ▼ ssh host …/bin/drishti-agent │ ▼ Exec format error

Fix shape (post-review)

Headline change vs. the original sketch: the registry takes a resolveDrvPath: (host) => Promise<string> callback, not a Record<system, drvPath> map. The map lives inside the callback's closure in main.ts; the registry stays ignorant of arches and systems.

1. flake.nix — one source for "supported systems"

Derive the per-system map from the existing systems list (flake.nix:19), don't hand-build it:

agentDrvBySystem = pkgs.lib.genAttrs systems
  (sys: self.packages.${sys}.drishti-agent.drvPath);

Rename the agent attr later → one place to update, Nix's evaluator enforces it.

2. default.nix:70 — JSON, not scalar

--set-default DRISHTI_AGENT_DRVS_JSON "${builtins.toJSON agentDrvBySystem}"

Without this, any TS change is invisible to nix run.

3. packages/app/src/server/archMap.ts — one mapping table

// `uname -ms` output → nix-system. Lives in drishti for now;
// migrate to @kolu/surface-nix-host when a second consumer appears.
export const UNAME_TO_NIX_SYSTEM = {
  "Linux x86_64":   "x86_64-linux",
  "Linux aarch64":  "aarch64-linux",
  "Darwin arm64":   "aarch64-darwin",
  "Darwin x86_64":  "x86_64-darwin",
} as const;

export async function resolveSystem(host: string): Promise<string> {
  if (isLocalHost(host)) {
    // map process.arch+process.platform → nix-system
    …
  }
  const out = await sshCapture(host, ["uname", "-ms"]);
  return UNAME_TO_NIX_SYSTEM[out.trim()] ?? throwUnsupported(out);
}

Reuses kolu's isLocalHost (host.ts:6-8) so we don't accumulate a fourth localhost-check.

4. main.ts:53-60 — parse map, build callback

const drvs = JSON.parse(process.env.DRISHTI_AGENT_DRVS_JSON ?? "");

const resolveDrvPath = async (host: string) => {
  const sys = await resolveSystem(host);
  const drv = drvs[sys];
  if (!drv) throw new Error(`no agent .drv baked for system=${sys}`);
  return drv;
};

const registry = await buildHostRegistry({ initialHosts, resolveDrvPath, … });

5. hostRegistry.ts — registry forgets that arches exist

// before
interface HostRegistryOptions { drvPath: string; … }
const buildEntry = (host: string): HostHandle => {
  const session = getHostSession({ host, drvPath: opts.drvPath, binary: "drishti-agent" });
  …
};

// after
interface HostRegistryOptions { resolveDrvPath: (host: string) => Promise<string>; … }
const buildEntry = async (host: string): Promise<HostHandle> => {
  const drvPath = await opts.resolveDrvPath(host);
  const session = getHostSession({ host, drvPath, binary: "drishti-agent" });
  …
};

buildHostRegistry itself flips to Promise<HostRegistry> — the initial seeding loop now awaits per-host probes. Parallelize:

const entries = await Promise.all(
  opts.initialHosts.map(async (host) => [host, await buildEntry(host)] as const)
);

Ripple stops at one await in main.ts (main.ts:80).

6. justfile — dev recipe has the same bug

The dev recipe today probes uname -ms on only the first positional host and bakes that single arch. With the new TS path doing per-host resolution, the recipe's probe goes away entirely — it just exports DRISHTI_AGENT_DRVS_JSON from the flake and lets the server resolve.

Why this shape (what reviewers changed)

concernfix
Registry shouldn't know "system" exists (Lowy) resolveDrvPath: (host) => Promise<string> callback, not a system→drv map in the API.
buildEntry complecting three concerns (Hickey) Probe + map + lookup all live in the callback constructed in main.ts; buildEntry becomes a two-liner.
Two localhost code paths drifting (Hickey) Reuse kolu's isLocalHost; one gate inside resolveSystem, not two parallel branches.
systems declared twice (Hickey) genAttrs systems over self.packages.${sys} — one source of truth for supported targets.
Justfile's first-host-only probe (Lowy) Same change, same PR — the recipe's bash probe goes away.

What didn't land

  • Moving the arch decision into kolu — kolu has neither the flake-ref nor the attribute-path, and its docstring explicitly delegates this (nixCopy.ts:7-13). A kolu-arch helper for the mapping table is a reasonable follow-up when a second consumer arrives.
  • Per-host arch caching — premature; one ssh uname -ms per add() is cheap, and the arch can't change without a host re-add.
  • A SystemDrvMap wrapper class — data is data.

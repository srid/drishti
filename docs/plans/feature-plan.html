1 Unified multi-host fleet dashboard

A single "pane of glass" aggregating every connected host into one live overview — the strategic bet.

Why it wins drishti's single most-validated differentiator. The multi-host plumbing (per-host agents, admin surface, reactive collections) already exists — this adds an aggregation / overview view, not new transport. It's what makes people choose drishti over a terminal htop.
Evidence of demand Netdata deliberately engineers parent-child centralization to "receive, store, process observability data from multiple systems" and present "unified dashboards across all your systems," scaling to thousands of nodes. Glances ships a "Central Browser" for exactly this. Single-pane-of-glass aggregation is corroborated as the recognized core benefit of multi-host monitoring across Datadog, logz.io, and eG Innovations.
Caveat Do not promise lossless cross-host resolution at arbitrary scale (that specific claim was refuted 1-2). Decide target scale first: 5–10 dev hosts (lightweight live aggregation suffices) vs. hundreds (needs streaming/persistence architecture).

2 Per-interface network I/O metrics

rx/tx per NIC and throughput — best effort-to-value ratio of the five.

The gap A concrete, well-scoped hole. drishti shows CPU, mem, command, cwd, load, and uptime — but zero network.
Evidence of demand rtop — drishti's closest conceptual peer (agentless, SSH-based, top-like, zero remote install) — defines its value around "vital system metrics (CPU, disk, memory, network)" including per-interface rx/tx. Glances treats network as baseline.
Implementation Read /proc/net/dev on Linux, netstat/sysctl on macOS through the existing agent, feed the existing reactive transport.
Difficulty: Low–Medium Confidence: High 2 supporting claims · both 3-0
  • rtop-monitor.org
  • github.com/rapidloop/rtop
  • github.com/nicolargo/glances

3 Historical retention + per-second time-series charts

A rolling time-series store per metric with sparkline/trend visualization.

The gap drishti shows only live state with no persistence. History unlocks "what happened 30 seconds ago" troubleshooting.
Evidence of demand Per-second collection + historical retention are Netdata's headline differentiators (dbengine retains ~14 days per-second by default). Source inspection confirmed drishti has no historical storage at all. The per-second cadence aligns naturally with drishti's reactive transport.
Decision needed Ephemeral in-memory ring buffer (preserves zero-config, low effort) vs. persistent with Prometheus/InfluxDB export (Glances exports to 18+ backends — but that's a larger second step).
Difficulty: Medium Confidence: High 3 supporting claims · all 3-0
  • github.com/netdata/netdata
  • github.com/nicolargo/glances

4 Container monitoring (Docker / Podman / LXC)

Per-container CPU/mem/IO surfaced alongside host processes.

The gap A competitive baseline drishti doesn't touch — and modern dev/ops machines are overwhelmingly containerized.
Evidence of demand Glances ships this as a core, documented capability (docker-py, podman-py, pylxd) with per-container CPU/mem/diskio/networkio.
Implementation Agent queries the runtime socket/API or reads cgroup files over the existing SSH agent; the UI extends the existing process-table model. Lower demand-urgency than 1–3, but high competitive-parity value.
Difficulty: Medium Confidence: High 4 supporting claims · all 3-0
  • github.com/nicolargo/glances

5 Richer in-UI process mutation — start with renice

Process-priority adjustment, beyond today's signal/kill.

Why Today the only mutation is signal/kill. renice is the smallest possible extension of a path that already proves drishti can run privileged actions safely over the SSH agent.
Evidence of demand btop users explicitly request in-UI renice, describing the friction of "copy PID → open terminal → run renice." Corroborated by action: btop PR #1268 (merged Oct 2025) added a renice menu, maintainer-approved.
Caveat Narrowest demand signal of the five (rests on one tool's issue tracker). Before piling on more mutations (cpulimit, ionice, STOP/CONT), weigh the added privilege surface.
Difficulty: Low Confidence: High 1 supporting claim · 3-0

Cross-cutting: protect the zero-install onboarding

drishti's zero-install-over-SSH agent + open-source posture are proven adoption accelerants (validated against rtop's "no special software needed on the remote" model and Netdata's zero-config auto-discovery). The implication for all five features: preserve zero-config onboarding — auto-discover hosts, containers, and NICs with no manual remote setup. These features should compound the low-friction story, not add setup burden.

Recommended build order

Quick wins first to build credibility, then the strategic differentiator, then the heavier features.

2 · network → 5 · renice → 1 · fleet dashboard → 3 · history → 4 · containers

Honest caveats from the research

  • All demand evidence is from competitor tools, not drishti's own users — a strong category-demand signal, but an indirect proxy. The biggest open question: what does drishti's own issue tracker actually request?
  • 5 claims were killed by adversarial verification — notably specific PLG funnel stats ("68% abandon trials," "10-min time-to-value → 3–4x conversion," "peer recommendation dominates discovery"). Do not anchor prioritization on those numbers.
  • The multi-host aggregation feature should not promise lossless cross-host resolution at arbitrary scale (refuted 1-2).

Open questions

  • What does drishti's own issue tracker / user community actually request most?
  • For multi-host aggregation, what scale must drishti realistically support — a handful of dev hosts, or hundreds of fleet nodes? This determines lightweight live aggregation vs. parent-child streaming.
  • Should historical retention be in-memory/ephemeral (preserves zero-config) or persistent with external export (InfluxDB/Prometheus)?
  • Beyond renice, which additional process mutations (cpulimit, ionice, OOM-score, freeze/STOP-CONT) carry enough demand to justify the added privilege surface?

Host stuck in “Connecting…” forever

Diagnosis from this machine’s live drishti.err.log, plus the fix and what the user should see. talk · --html

What’s actually happening (your screenshot, your logs)

vanjaram is wedged in connecting right now. The log shows the healthy link drop with SSH code=255, the reconnect loop re-arm — and then it spawns, the remote agent starts serving, and the trail just stops:

[vanjaram local] connection: connected → disconnected [vanjaram local] lastError: agent exited (code=255, signal=null) [vanjaram local] reconnecting in 4000ms… (attempt 2/5) [vanjaram local] connection: copying → connecting [vanjaram remote] serving surface over stdio (read=stdin, write=stdout) ⟨ … nothing. no "first RPC received". no "agent exited". forever. ⟩

Contrast the healthy spawns just above it, which always log first RPC received — link is live → connecting → connected. The stuck spawn never gets there: the SSH child is alive (transport up, agent serving), but the first RPC never roundtrips and the process never exits.

Why nothing recovers it

The connection state machine lives in the dependency @kolu/surface-nix-host (node_modules/@kolu/surface-nix-host/src/hostSession.ts). Every failure path is guarded by a timer — except the one you hit.

copying ──provision ok──▶ connecting ──first RPC──▶ connected
                     connecting ──child exit/error──▶ disconnected ──retry──▶ failed

The only exits from connecting are:

  • Success: markConnected() — fired by drishti’s router after the first system snapshot yields. router.ts:365 → hostSession.ts:319
  • Failure: child.on("exit") / child.on("error") → scheduleReconnect(). hostSession.ts:285, :294, :344
The gap: spawn() sets connecting at hostSession.ts:271 and arms no timeout. A child that stays alive but never completes the RPC handshake fires neither exit. markConnected never runs. The 5-attempt give-up gate is downstream of scheduleReconnect, which is never reached. Stuck in connecting with no escape.

drishti can’t paper over this from its side: session.reconnect() is guarded if (this.clientPromise !== null) return — non-null during connecting, so it no-ops — and there’s no exposed kill. hostSession.ts:336 The fix belongs in @kolu/surface-nix-host.

Fix: a connect-phase watchdog (post-review design)

Symmetrical with the reconnect timer that already exists. On entering connecting, arm a timeout; if markConnected hasn’t fired by then, kill the child and let the existing exit path route it through scheduleReconnect — so it counts toward the give-up gate and lands in the terminal failed state, reusing all of #27’s UI.

Both the Lowy (boundary) and Hickey (structure) passes ran on the first sketch; the design below already folds in what landed.

DecisionShape (after review)
Where Inside HostSession — same volatility (“link lifecycle & failure”) that already owns reconnectTimer. Not in drishti’s router, which would have to learn the connecting phase, timer lifetime and teardown. hostSession.ts:99,344
One timer, not two Hickey F1. A session has at most one pending phase-transition trigger — you wait for the reconnect delay or the connect watchdog, never both. Don’t add a second connectTimer field beside reconnectTimer; collapse both into one pendingTimer + armTimer(ms, fn) / clearTimer(). The two callbacks stay distinct — only the slot is shared, so this de-duplicates without complecting timeout with backoff. Update the double-arm guard at :345 to the unified field.
Clear in one place Hickey F2. Don’t smear clearTimeout across four transition handlers. Every exit from connecting already routes through updateState() (:203) — clear the timer there when patch.connection !== "connecting". One mechanical choke-point; teardown() keeps its explicit clear since it isn’t a state transition.
On fire: kill only Lowy + Hickey P3. The callback sets lastError = "connect handshake timed out after Ns" and calls child.kill("SIGTERM") — nothing else. kill is async, so the existing child.on("exit") handler (:285) fires next tick and does the scheduleReconnect / bookkeeping. Inlining that in the callback would duplicate the body and double-arm the retry. Guard the exit handler to preserve a pre-set lastError instead of overwriting it with the signal string. Add a top-of-callback state === "connecting" check for the markConnected-just-fired race.
Timeout value Lowy. Surface connectTimeoutMs?: number on HostSessionOptions as a sibling of reconnectDelayMs (:69), default ~30s — a high-latency-VPN consumer or a fast-fail test harness shouldn’t have to fork the module. (MAX_CONSECUTIVE_FAILURES stays a constant — nothing varies it.)
Shared failure counter? Yes — folding the timeout into consecutiveFailures is correct dedup, not conflation. A host that consistently hangs its handshake is as broken as one that consistently crashes; the 5-attempt ceiling fits both. hostSession.ts:355

Net: one field, two helpers, one choke-point clear, ~30 lines in kolu — and drishti passes connectTimeoutMs at hostRegistry.ts:91. P3 wants a /fact-check on the real diff to confirm the kill→exit ordering.

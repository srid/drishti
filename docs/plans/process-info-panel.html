Process info panel

Click a process row to open a detail panel. Design + delivery record.

Status: phases 1 & 2 shipped in the feat/process-info-panel PR. Phase 3 (per-process sparkline) deferred to issue #21.

Motivation

The process table already renders every field a Process carried (user, cpuPct, rssBytes, command, cwd), but it truncates the command (CSS max-w-md truncate) and crams cwd inline after an @. There was no way to see a process's full command line, its full working directory, or any of the metadata an operator reaches for when triaging (parent, state, niceness, threads, how long it's been up). The feature is two complementary slices:

  • Phase 1 — a click-to-open detail panel that shows the existing fields in full (untruncated command + cwd, exact CPU% and resident bytes, plus the process's share of host RAM).
  • Phase 2 — enrich the Process surface with five new fields so the panel earns its place beyond "a wider row".

linux fields come for free from the /proc/<pid>/stat read the agent already does. darwin only gains ppid/nice/state (all standard BSD ps keywords); threads/startedAtMs are null there — an unknown ps keyword would make the whole command fail and drop every process, so we don't reach for one we can't verify. null (not 0) keeps "unavailable" distinct from a real value at the type level.

Design decisions

Selection state lives in HostView, reached by rows via context

selectedPid is an ephemeral createSignal<Pid | null> in HostView — the same tier that owns the processes store and the filter/sort prefs. It is not persisted (a transient focus, not a setting). ProcessRow reaches it through a small SelectionContext for its click + highlight, so ProcessTable stays a pure renderer with an unchanged signature — mirroring how the app-level view signal threads down without widening every component's props.

The panel is a sibling of the table, fed the resolved Process

ProcessDetail renders as a <Show> sibling above the table (not an overlay — no z-index/focus-trap machinery), mirroring the app's <Show when={selectedHost()}>. Its prop is the resolved process: Process, so the phase-2 fields flowed in with no signature change — phase 1 and phase 2 met at this boundary.

Selection resolves against the live set, cleared at the source

A selected process can exit. Two guards, each covering a different path: the delta removes loop clears selectedPid at the source (also avoiding stale-pid resurrection under pid recycling), and a selected memo resolves processes[pid] reactively so a process dropped by a full snapshot (which doesn't go through the removes loop) also hides the panel. Same "resolve focus against the live set" shape as the app's resolvedView.

Change detection by explicit field list

The agent poll loop only re-publishes a row when a wire-visible field changed. This was a hand-maintained OR-chain; the feature replaced it with an explicit MUTABLE_PROCESS_FIELDS list compared via .some(), satisfies (keyof Process)[] tying each entry to the schema. startedAtMs is deliberately excluded — it is immutable per pid, so it never gates a publish regardless of how the agent derives it.

Phase 3 — deferred

A per-process CPU/mem sparkline in the panel needs per-PID history retention in the parent (today's metricHistory ring is system-wide, {t, cpu, mem}, no PID). That's a backend lift — bounded per-PID rings with eviction under churn — tracked in issue #21, not part of this PR.

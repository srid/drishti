“localhost” timed out — the host that needs no connection

Why the one tab that should always be green is the only one that’s red. talk · --html

The punchline. You’re right that localhost uses no network — and it doesn’t. But “no connection” isn’t “no handshake.” drishti treats every host uniformly: even localhost gets nix-store --realise → spawn the agent as a local stdio child → wait ≤30 s for the first RPC. The agent enumerates all 956 processes before it starts answering, and on zest — a load-9.58 macOS box running lsof over its own process table — that enumeration overruns the 30 s budget. The host that “should just work” is in fact the worst case in the fleet.

What you’re looking at

The card says connect handshake timed out after 30000ms (transport up, no first RPC), then it backs off and retries to attempt 5/5, then gives up. The log shows the agent realising fine and only then the timeout:

[local] connect handshake timed out after 30000ms (transport up, no first RPC) [local] reconnecting in 16000ms… (attempt 4/5) [local] localhost: realising '/nix/store/…-drishti-agent.drv'… [local] warning: you did not specify '--add-root'; the result might be removed by the GC [local] localhost: agent realised at /nix/store/…-drishti-agent

That ordering is the first clue: the realise succeeds (agent realised at …) — the 30 s is spent after it, on the handshake. The slow-nix-build theory is dead on arrival.

localhost really does skip the network

The transport layer special-cases the loopback names — no ssh, no nix copy:

So your instinct holds: nothing leaves the box. What doesn’t change for localhost is the session lifecycle — it still spawns a child and waits for a handshake under a watchdog:

copying ──realise locally (outside the 30s)──▶ connecting ──first RPC──▶ connected
                         connecting ──30s watchdog──▶ disconnected ──×5 (cause “remote”)──▶ failed
The 30 s watchdog is armed the instant the child is spawned — after provisionAgent (probe + realise) returns. So realise time is genuinely not in the budget; the whole 30 s belongs to the agent’s own post-spawn startup. watchdog armed at spawn: hostSession.ts:515 · default 30s: hostSession.ts:433 · wired by drishti: hostRegistry.ts:46,117

Why it times out anyway: the agent serves last

The “first RPC” the parent waits on is the bridge’s system.get subscription — its first yield calls markConnected() (connecting → connected). But the agent can’t answer until it reaches serveOverStdio(), and that is the last thing main() does:

main.ts:106 await reader.readSystem() // darwin: vm_stat + statfs (cheap) main.ts:110 await reader.readProcesses() // darwin: ps + lsof -nP -d cwd ← all 956 pids main.ts:120 await reader.readNetwork() // darwin: netstat -ib … main.ts:300 await serveOverStdio(…) // ← only NOW is stdin read / RPCs answered

Those are plain sequential awaits, not a background warm-up. The parent’s system.get frame is written to the child’s stdin the moment the link opens — and then sits buffered, unread, because serveOverStdio hasn’t attached the stdin reader yet. The dominant cost is lsof -nP -d cwd over the whole table (the per-process cwd lookup added in 3d6ff0b). On a load-9.58 box with 956 processes that overruns 30 s → watchdog → SIGTERM → the exit handler reports the verbatim string and classifies it "remote" (bounded), so after 5 strikes it lands in terminal failed.

handshake = system.get first yield → markConnected: router.ts:404–411 · ps+lsof: proc.ts:681 · timeout string + cause "remote": hostSession.ts:473 · give-up after 5: hostSession.ts (MAX_CONSECUTIVE_FAILURES)

The irony: localhost is the worst host, not the easiest

Three things stack up, and they all land hardest on exactly this host:

  • localhost is zest. The agent runs on the most-loaded, highest-process-count box in the fleet — and competes for the very CPU it’s trying to measure.
  • It’s darwin. macOS has no /proc, so the reader forks ps + lsof + netstat. Your green Linux remotes use pure /proc file reads (proc.ts:328–381) — no lsof fork — so they handshake sub-second even with more PIDs.
  • The agent serves last. All 956 PIDs of lsof sit squarely on the handshake critical path, instead of filling in a beat after the link is live.

A remote Linux host with 200 idle processes connects instantly; localhost — “which needs no connection” — is the one whose agent is too busy enumerating its own process table to say hello in time.

Fix: serve first, enumerate lazily

The handshake only needs the cheap system cell — already seeded by readSystem() at main.ts:106. It does not need the lsof-driven process table or netstat. Reorder so the agent answers RPCs immediately and lets its existing 2-second poll loop fill the rest.

DecisionShape
Move the seam Call serveOverStdio() right after the cheap readSystem() seed; defer the two subprocess-backed reads (readProcesses’ lsof, readNetwork’ netstat) to the first tick() of the existing poll loop. main.ts:105–121 vs :300
Keep the cheap seed Don’t defer everything. readCpuCores() (main.ts:115) is synchronous os.cpus() — keep seeding it pre-serve; it costs nothing and gives the CPU-delta its baseline. Only the fork-backed reads belong off the critical path.
Uniformity preserved No localhost special-case. This is an agent-startup fix, not a transport branch — so it speeds up every host’s cold start, darwin or linux, local or remote. The deliberate “localhost is not special” design (hostRegistry / host.ts) stays intact.
Accept the first-frame blip CPU% / net throughput are deltas against a prior tick (proc.ts:64–95, 201–220); a browser that subscribes in the sub-second gap before tick #1 sees one frame of 0% / 0 B/s and a momentarily empty process list, corrected on the next delta. Benign under the snapshot-then-delta contract the pumps already tolerate.
Not the watchdog Raising / scaling connectTimeoutMs is a band-aid: it hides the latency instead of removing it, and a 956-proc darwin box under heavier load would still blow a bigger budget. Leave the 30 s where it is.

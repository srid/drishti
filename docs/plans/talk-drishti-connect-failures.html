Why the 3 hosts fail to connect

Production drishti (launchd, home-manager) on this machine · log ~/Library/Logs/drishti.err.log · server pid 75024, up 3h, LastExitStatus=0 (not crash-looping).

It is not ssh, not trusted-users, not the agent copy. ssh works, the closure copies, the agent re-realises and runs perfectly on every attempt. The failure is entirely in the parent server's per-host reconnect bridge: after the first ssh link drops, the parent stops issuing the system.get subscription to the freshly-respawned agent. The agent boots, prints serving surface over stdio, and waits for a first RPC that never arrives — so kolu's 30 s connect-watchdog kills it, 5 strikes → failed. The "Copying agent to remote…" you saw is just one frame of that retry spin.

So "I can ssh fine" is fully consistent with the symptom: ssh, the closure, and the agent are all healthy. What's broken is the parent's in-process handoff from "the agent respawned" to "re-issue the subscription on the new client." CONNECT_TIMEOUT_MS = 30_000 (hostRegistry.ts ~:40) is doing its job — raising it would only delay each kill, not deliver the missing RPC.

What the logs can't tell us yet — and why we need richer logging

Two things blind the diagnosis from here, both fixable:

  • The [bridge] prefix carries no host tag. All five hosts' bridge loops write a flat [bridge] … (router.ts log() ~:264), so their lines interleave indistinguishably. kolu already tags its own lines [host:X local|remote] — drishti's bridge breaks that symmetry.
  • No timestamps anywhere. Can't measure the 30 s boundary, can't tell a slow first-snapshot apart from a never-sent subscription, can't correlate the ssh drop with the respawn.

With those two gaps closed, the next production log will say outright which of the two handoff failure modes is biting (bridge latches a stale/dead client vs. system.get on the new client hangs without erroring).

The reconnect-handoff gap stands.

"Isn't it time for a real logging library?" — No.

I came into this leaning toward pino (child loggers for per-host context, free timestamps/levels/JSON). Both reviewers pushed back, independently and convincingly, and I've reversed. Here's the honest reasoning, because the instinct is reasonable but wrong for this codebase:

  • The problem is fragmentation, not feature-poverty. There aren't "no features" — there are five hand-rolled one-line writers, each with its own ad-hoc bracket prefix: [server] (main.ts:39), [hosts] (hostsStore.ts:41), [bridge] (router.ts:265), [admin] (admin-router.ts:77), and an unprefixed one in the agent (agent/main.ts:53). That's one concept split five ways. The fix is to stop splitting it — a 3-line factory — not to import a framework.
  • pino's structure answers no consumer drishti has. Logs are read in a journal / tmux pane, not an aggregation pipeline. JSON-in-a-terminal would itself need a prettifier (another dep) to be readable. The project's whole ethos is zero-runtime-dep, npins, "zero config" — a logging dependency is a poor trade when the only structurally-useful pino feature (child({host}) context) is reproducible in one closure.
  • A library wouldn't even cover the lines that matter. kolu writes its [host:X local|remote] lines straight to process.stderr; no logger you add to drishti restructures those. Timestamps — the one thing you genuinely lack — have to come from the sink, not the writer (below). A logger in drishti would produce a JSON/plaintext mixed stream, strictly worse to read than uniform plaintext.

Verdict: a tiny makeLogger(tag) factory + per-host context + sink-level timestamps gets everything a library would give that drishti can actually use, at zero dependency cost. The plan below is the post-review shape.

The actual plan to deploy (reviewed — Lowy + Hickey)

All drishti-side; no edits to the pinned kolu library.

1 · One makeLogger(tag) factory, host-scoped at the bridge

New server/log.ts: makeLogger(tag) => (line) => process.stderr.write(`[${tag}] ${line}\n`). Replace all five writers — including the two inline process.stderr.write sites a naive swap would miss: hostsStore.ts:41/:48 and admin-router.ts:77 (both load-bearing error paths — corrupt hosts-file, failed host-removal). Then the load-bearing fix: buildRouter builds makeLogger(`bridge:${host}`) and threads the logger (not the raw host) into bridgeAgentToParent + the four pumps — which already accept a log param (router.ts ~:322/:339), so internals don't change. host is already at the call site (hostRegistry.ts buildEntry). This is the same per-host attribution child({host}) would give, minus the dependency — and it fixes the real [bridge]-interleaving bug, not just cosmetics.

2 · Timestamps at the sink — covers kolu too

A timestamp is a property of when a line hits the sink, so the factory deliberately does not stamp (that would miss kolu's own [host:X] writes and the forwarded agent lines). Time the unified stream once:

  • linux: free — journald already stamps (journalctl --user -u drishti).
  • darwin (your prod): pipe stderr through moreutils ts '%FT%T.' in nix/home/module.nix:99 (StandardErrorPath). No code, no global monkeypatch, covers drishti + kolu + forwarded-agent lines uniformly.

Reviewers split on whether to also re-emit kolu's structured onState data (connection/lastError/progressLines, already consumed at router.ts:183) through drishti's logger. Resolution: don't — kolu's stderr already carries that content; re-logging it would double every line and invent a "cell and stderr must agree" invariant. Consume the cell for the UI (as today); leave logging to kolu's own stderr + the sink timestamp. No kolu onLog upstream needed.

3 · Instrument the reconnect handoff — in drishti's pump (the real point)

The blind spot is the gap between "pumps started" and "first snapshot" inside pumpSystemCell (router.ts): if the subscription never fires, nothing logs. On the now-host-tagged bridge logger, add:

  • a per-spawn client sequence id at agent client ready; starting pumps;
  • issuing system.get subscription (client #N) immediately before the for await … client.surface.system.get({});
  • elapsed-ms-to-first-yield in the existing n === 1 branch.

If #N climbs but "issuing" never reaches a yield → the subscription hangs on the new client; if "issuing" never logs for #N>1 → the bridge never got the new client. Either way the next log pins which line to fix.

4 · Agent-side liveness (optional, confirms the idle-wait)

In agent/main.ts: pid in first RPC received, and a waiting for first RPC (Ns)… heartbeat while serveOverStdio blocks on stdin. A respawned agent ticking to ~30 s with no RPC is the proof from the remote end.

Bottom line

The three hosts aren't failing to connect — they connect once, then the parent's reconnect bridge fails to re-subscribe after the first ssh blip, and the watchdog reaps the idle agents. The fix lives in bridgeAgentToParent/pumpSystemCell (and possibly kolu's waitForNextClient handoff), but ship the logging enrichment first — items 1–3 are ~an hour, touch only drishti, and the next production log tells us exactly which handoff line to fix instead of guessing.

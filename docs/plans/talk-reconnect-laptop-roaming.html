Reconnection on a roaming laptop

You asked what the reconnect logic looks like and how to improve it — specifically for the close-the-lid-at-home, reopen-at-the-coffee-shop workflow. Here is what's verified in the source today, why that workflow lands a host in the terminal failed state, and the ranked fixes — across kolu, drishti, and the browser.

TL;DR. drishti's parent ssh's from your MacBook to each host. When the lid closes that ssh link dies, and kolu's reconnect loop gives up after 5 attempts in ~30 s — which is shorter than the time café Wi-Fi takes to come up — so every host lands in terminal failed and needs a manual Reconnect click. The gate is built for config faults (auth) but punishes network blips. Fix, in order of leverage: (1) classify failures so network faults never terminate — they just keep retrying and self-heal when the network returns (kolu; the one that matters); (2) a wake detector that re-arms every host ~1 s after the lid opens instead of waiting ~30 s (kolu/drishti); (3) browser online/refocus nudges, (4) tighter ssh keepalive, (5) drop stale green sooner (polish). If you do only one thing, do (1).

The reconnect logic today — two independent loops

"Reconnect" is two different loops that share one word. On a laptop they live on opposite sides of the sleep boundary:

LoopOwnerBehaviourSurvives sleep?Source
browser ↔ parentpartysocket Reconnects forever, 2–15 s backoff. Yes — it's loopback (ws://localhost). wire.ts:30-36
parent ↔ agentHostSession (kolu) ssh subprocess. Backoff 2→4→8→16 s (cap 60), then gives up after 5 → terminal failed. No — the ssh TCP socket dies the moment you roam networks. kolu hostSession.ts:115,481-508

drishti runs as a parent process on the MacBook and ssh's out to each host; the browser talks to it over loopback. So the fragile link is always parent → agent, and the give-up gate on that link is the whole story.

The relevant constants, verified in the pinned kolu (npins → 66be612e)

  • MAX_CONSECUTIVE_FAILURES = 5 → after the 4th retry the session sets connection:"failed" and stops. hostSession.ts:115,486-502
  • Backoff min(2000 · 2^n, 60000) = 2 s, 4 s, 8 s, 16 s, then give up. hostSession.ts:505-506
  • connectTimeoutMs = 30 s handshake watchdog: ssh child alive but no first RPC ⇒ kill & count as a failure. hostRegistry.ts:46 → hostSession.ts:374,431
  • The long-lived agent ssh carries ServerAliveInterval=10 — but no explicit ServerAliveCountMax, so ssh's default of 3 means a dead peer takes ~30 s to notice. kolu host.ts:44-54
  • consecutiveFailures resets to 0 only on markConnected() (first RPC) or a manual reconnect(). hostSession.ts:447,465-468
  • Nothing listens for OS wake, network change, online/offline, or tab refocus on either side. (Grep: no powerMonitor/online/visibilitychange driving reconnection anywhere.)

What actually happens when you close the lid

10:00 vanjaram connected — agent streaming, ssh channel warm (ServerAliveInterval=10) 10:00 ── lid closes · macOS suspends the drishti parent process · all timers freeze ── 12:30 ── lid opens at the café · new SSID · captive portal · no route yet ── 12:30 any pending reconnect setTimeout fires immediately on wake → spawn ssh → no network → fail (attempt 1) 12:30 backoff 2s → fail (2) → 4s → fail (3) → 8s → fail (4) → 16s → gave up after 5 12:31 vanjaram failed — terminal. User must click Reconnect. Per host. Every single time.
The give-up gate is calibrated for the wrong failure. Five attempts over ~30 s makes sense for a configuration fault (trusted-users, a missing ssh key) — retrying won't fix a misconfiguration, so giving up and surfacing the error is right. But a roaming laptop's disconnect is a transient network fault: the café Wi-Fi needs 20–40 s for the captive portal + DHCP + DNS to settle. The backoff burns all five attempts inside that window, so the host is declared permanently failed before the network is even usable — and the one workflow drishti is most used in (laptop, lid, café) is the one where it always needs a manual rescue.

Two secondary papercuts ride along:

  • Stale green. Before the gate even engages, the still-"connected" ssh child takes up to ~30 s (ServerAliveInterval=10 × default count 3) to notice the dead socket. For that window the UI shows a healthy emerald dot while every RPC silently hangs.
  • Sleep eats a retry. A reconnect timer pending at sleep fires the instant the process resumes — typically before the new network has a route — wasting one of the five attempts on a guaranteed miss.

How to improve it — ranked

1Don't treat a network blip as terminal kolu — the big one

Classify the failure and let the policy depend on the class. The failureCause discriminant was already proposed as the right seam in the earlier reconnect work (talk-reconnect-noop.html); this is its strongest motivation:

// kolu hostSession.ts — at each failure site, tag the cause
type FailureCause = "network"    // ssh code 255, connect/handshake timeout, ServerAlive death
                   | "config"     // auth refused, signing failed, trusted-users, nix copy denied
                   | "unknown";

// give-up gate becomes cause-aware
if (cause !== "network" && this.consecutiveFailures >= MAX_CONSECUTIVE_FAILURES) {
  this.clearClientPromise();
  this.updateState({ connection: "failed" });   // config faults still terminate — correct
  return;
}
// network faults never terminate: keep cycling disconnected→connecting at the 60s cap, forever

Net effect: a café-roam host sits in disconnected ("reconnecting…", amber, honest) and self-heals the moment the network returns — no click. failed goes back to meaning what it should: a human has to fix something. This is the single change that fixes the headline workflow.

2Wake detector — reconnect within a second, not 30 kolu / drishti

Don't wait for ssh's ~30 s keepalive to discover a dead link after wake — detect the wake itself. A pure-Node, cross-platform trick: a 1 s interval that watches for a wall-clock gap. If the loop was asleep far longer than its period, the machine suspended:

// runs in the parent — no native deps, works on macOS/Linux
let last = monotonicNow();
setInterval(() => {
  const now = monotonicNow(), gap = now - last; last = now;
  if (gap > 5_000) onWake(gap);   // expected ~1s; a big gap == we were suspended
}, 1_000);

// onWake: for every pooled session — reset the gate, drop the stale child, respawn now
for (const s of registry.sessions()) s.reconnect();   // already resets consecutiveFailures + re-arms

This collapses the "stale green for 30 s, then maybe give up" window into "reconnects ~1 s after the lid opens." Pairs with #1: even on a slow café network, #2 kicks the first attempt immediately and #1 keeps it trying until the portal clears. reconnect() already exists and does exactly the reset+respawn we need (hostSession.ts:465-468) — the only new piece is the wake signal and a way to fan it across the pool. The clean home is kolu (the registry owns the session set and the lifecycle volatility); drishti would only own it if we'd rather keep the OS-signal concern out of the library.

3Browser online / refocus → re-probe browser — cheap belt-and-suspenders

The browser tab is literally the thing the user reopens. partysocket already reconnects its loopback link, but we can also nudge the parent to re-probe agent links on the events the OS hands us for free:

// client — on regaining connectivity or refocusing the tab, ask the parent to re-arm all hosts
addEventListener("online", kickReconnectAll);
document.addEventListener("visibilitychange", () => {
  if (document.visibilityState === "visible") kickReconnectAll();
});
// kickReconnectAll → adminClient.hosts.reconnect({host}) per host (RPC already exists)

Reuses the existing hosts.reconnect admin RPC (admin-router.ts:86-93, App.tsx:608-620). Independent of #1/#2 and safe to ship alone, but it's a hint, not the cure — if a host already hit terminal failed without #1, this re-arms it, which is a nice bonus.

4Bound the dead-link detection explicitly kolu

The agent ssh sets ServerAliveInterval=10 but leaves ServerAliveCountMax at ssh's default 3 (~30 s). For plain network drops where #2's wake signal doesn't fire (weak café Wi-Fi that flaps without a suspend), pin it:

// kolu host.ts:44-54 — make the worst-case detection time explicit and shorter
"-o", "ServerAliveInterval=10",
"-o", "ServerAliveCountMax=2",   // ~20s worst case instead of ~30s; documented, not implicit

Minor on its own; matters as the floor for non-sleep flaps. Keep it modest — too aggressive and a brief latency spike tears down a healthy link.

5UI: don't show green during the stale window drishti

After a detected wake (or on the first hung RPC), flip the dot to amber "reconnecting…" rather than leaving the stale emerald until the link is reconfirmed. Small honesty fix layered on the existing connectionColors state map (connectionColors.ts:36-78); the disconnected presentation is already amber+pulse, so it's mostly about entering that state sooner.

Where each change lands (Lowy boundary check)

Connection-lifecycle and failure-classification volatility belongs in kolu — it owns the state machine, the give-up gate, and reconnect(). drishti stays a thin mirror plus a couple of OS-signal nudges; the browser contributes only the events the platform already emits. A drishti-side workaround (string-matching kolu's log lines to guess "is this a network fault?", or destroy+recreate sessions to force a retry) would leak the state machine across the repo boundary — the same anti-pattern the earlier reconnect docs rejected.

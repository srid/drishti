Solving the reconnect UX

When a host can't be reached, the screen lies. Here's why, what's verified in the source, and the single coordinated fix across kolu and drishti.

What the screenshot is actually showing

The pureintent tab is red and the pane says "Disconnected. Retrying…". The log (/tmp/drishti.err.log) says otherwise for that exact host:

[host:pureintent local] reconnecting in 16000ms… (attempt 4/5)
[host:pureintent local] sign_and_send_pubkey: signing failed for ED25519 "id_ed25519" … agent refused operation
[host:pureintent] lastError: pureintent: 'nix copy --derivation' exited with code 1
[host:pureintent local] gave up after 5 consecutive failures — fix the underlying issue
        (often: remote nix-daemon needs your user in 'trusted-users' …) and restart the parent
The parent gave up — it is not retrying. The UI says "Retrying…" anyway, and the one fact that would let the user fix it (the ssh-agent / trusted-users auth failure) never leaves the log. A dead-end wearing a "working on it" label.

Two retry loops, conflated into one word

LoopWhoBehaviorSource
browser ↔ parentpartysocket Reconnects forever, 2–15 s backoff. Healthy in the shot.wire.ts:26-37
parent ↔ agentHostSession (kolu) Retries the ssh / nix copy link 5×, then gives up permanently. kolu hostSession.ts:330-335

The browser socket is fine and faithfully reporting disconnected. What died terminally is the parent→agent link. "Retrying…" is true of the wrong loop.

Root cause — verified in source

I read the pinned kolu (npins → commit 625dbeba). Two distinct gaps:

Gap 1 — drishti throws away what kolu already exposes

HostSessionState carries three fields, and onState hands all three to drishti:

// kolu hostSession.ts:43-50  — already on the wire-facing callback
interface HostSessionState {
  connection: ConnectionState;          // the 4-enum
  progressLines: readonly string[];   // last 20 lines incl. "reconnecting… (attempt 4/5)"
  lastError: string | null;            // the real error — ALREADY populated
}

But drishti's mirror keeps only connection and discards lastError + progressLines:

// drishti router.ts:183-184
session.onState((s) => {
  fragment.ctx.cells.connection.set({ state: s.connection }); // drops s.lastError, s.progressLines
});

So the real error and the attempt-counter text are already available at this one line — drishti just isn't forwarding them. No grep for lastError/gaveUp hits anywhere in packages/app/src/.

Gap 2 — kolu collapses "retrying" and "gave up" into one value

On give-up, kolu logs the message and returns without changing connection — it stays "disconnected", the same value as "between retry attempts". And the attempt counter (consecutiveFailures) is private:

// kolu hostSession.ts:330-335 — terminal give-up sets no terminal state
if (this.consecutiveFailures >= MAX_CONSECUTIVE_FAILURES) {
  this.addLocalProgress(`gave up after …`);
  return; // connection stays "disconnected" — indistinguishable from "will retry"
}
This is why drishti can't tell the truth with a drishti-only change: "still retrying" and "gave up forever" are the same connection value, and the attempt count is unexposed. Reconstructing the distinction by string-matching progressLines for "gave up after" would couple drishti's wire schema to kolu's log-line wording — the wrong seam. The terminal state belongs in kolu, next to the give-up gate that already owns it.

The fix — one coordinated change, two repos

kolu — make the give-up observable (≈10 lines, hostSession.ts)

// 1. add a terminal state to the enum (line 37-41)
type ConnectionState = "copying" | "connecting" | "connected" | "disconnected" | "failed";

// 2. set it on give-up — lastError is already populated (line 333)
if (this.consecutiveFailures >= MAX_CONSECUTIVE_FAILURES) {
  this.addLocalProgress(`gave up after …`);
  this.updateState({ connection: "failed" });
  return;
}

// 3. add a public re-arm for the Reconnect button (no such method exists today)
reconnect(): void {                       // reset the gate + respawn
  if (this.destroyed || this.refCount === 0) return;
  this.consecutiveFailures = 0;
  this.clientPromise = this.spawn();
}

Then bump the npins kolu pin in drishti. disconnected keeps its existing meaning ("link down, cycling through copying→connecting again"); failed is the new terminal concept. The attempt text ("attempt 4/5") rides along for free in progressLines — no new structured field needed.

drishti — mirror the richer state and render it

The wire cell becomes a faithful projection of HostSessionState rather than a re-modelled union — kolu owns the lifecycle vocabulary, so drishti stays a thin mirror with a single translation point (the seam both reviewers endorsed):

// surface.ts — mirror kolu's shape; lastError/progressLines are cross-cutting, not per-variant payloads
const ConnectionSchema = z.object({
  state: z.enum(["copying","connecting","connected","disconnected","failed"]),
  lastError: z.string().nullable(),
  progressLines: z.array(z.string()),   // optional; powers the live "attempt N/5" line
});

// router.ts:183-184 — forward, stop discarding
session.onState((s) => fragment.ctx.cells.connection.set({
  state: s.connection, lastError: s.lastError, progressLines: s.progressLines,
}));
  • connectionColors.ts — disconnected flips to amber + pending:true + "Reconnecting…" (honest); add failed = red, solid, terminal. The STATE map stays a clean Record<state, visual> for dot/pulse; the renderer interpolates lastError / the progress tail (so no dynamic strings get baked into the static map).
  • App.tsx ConnectingOverlay (952-962) — widen props from a bare state string to the full connection value; render failed as the error card below.
  • new admin RPC hosts.reconnect({ host }) → calls the pooled session's new reconnect(). Not reused add/remove — see below.

Two reviewer findings, folded in

Dedicated reconnect, not remove+add. Both reviewers killed my first idea of re-arming via the existing add/remove RPCs (App.tsx:386,397). Hickey: it races resolvedView's fallback (App.tsx:351-355) so the host visibly vanishes from the fleet, plus a double saveHosts. Lowy: re-arming a registered session is a different volatility from host-set membership. Verified in source that HostSession has no public re-arm — hence the new kolu reconnect() + a purpose-built hosts.reconnect verb.
Single translation point. Lowy's seam: normalize at the one onState site (router.ts:183-184), so a future kolu error-taxonomy change touches only that line. Mirroring kolu's flat {state, lastError, progressLines} shape (rather than inventing a discriminated union) keeps drishti a thin projection — the union would re-model vocabulary kolu already owns. Trade-off: the union is more precise (can't attach lastError to connected); the flat mirror is simpler and lower-coupling. Recommending the mirror.

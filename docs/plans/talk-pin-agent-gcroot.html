Pinning the realised agent against GC

Should the remote keep a GC root on the realised agent? And should that root be a moving “latest” link? — short answer: yes to both.

1. You’re right — there is a real gap

The provisioning path lives in @kolu/surface-nix-hostsrc/nixCopy.ts, and the realise step is: 

// step 2 — realise the .drv on the target
ssh $host nix-store --realise $drvPath

No --add-root. That matters because nix-store --realise only creates a temporary root under /nix/var/nix/temproots/<pid> that is held for the lifetime of that ssh process and no longer. The instant the realise ssh exits, nothing on the remote roots the agent’s output closure. A nix-collect-garbage / nix store gc on the remote — cron, a teammate, nh clean, a NixOS GC timer — is free to delete it.

What softens this today (but doesn’t fix it): the surface-nix-host session lifecycle is disconnected → copying → connecting, and the copying edge re-runs provisionAgent(). So a GC’d path gets re-realised on the next reconnect — the agent isn’t permanently broken. But that’s a rebuild (or re-substitute) on every reconnect after a GC, and during a live connected session nothing protects the closure. The running process keeps its inodes open so it survives mid-flight, but the store path is deletable and the next spawn pays to rebuild. The pin turns “rebuild-on-reconnect-after-GC” into “realise once, reuse forever.”

2. The fix is one flag — and it gives you “latest” for free

Add an indirect GC root at realise time:

ssh $host nix-store --realise --add-root $rootPath --indirect $drvPath
FlagWhat it does
--add-root PCreates symlink P → /nix/store/…-agent and registers it as a GC root.
--indirectRegisters the symlink itself via /nix/var/nix/gcroots/auto/<hash> → P. This is what lets P live in $HOME (user-writable) instead of needing root to write under /nix/var/nix/gcroots. The ssh user is almost never root, so --indirect is mandatory here, not optional.

The “latest link” semantics you described fall out automatically

Because the root is a single, fixed-name symlink, each new realise overwrites it to point at the new output. The previous output is no longer referenced by that root, so it becomes GC-eligible — exactly “pin the latest, unpin older hashes,” with no generation accumulation and nothing to prune. This is precisely how nix build --out-link result behaves; you’re just relocating that result link onto the remote.

I verified the mechanics locally before writing this:

# realise drv A with a named root
$ nix-store --realise --add-root /tmp/pintest/agent --indirect $drvA
/tmp/pintest/agent
$ ls -l /nix/var/nix/gcroots/auto/
…xhrm4… -> /tmp/pintest/agent            # auto-root registered

# realise a DIFFERENT drv B to the SAME link
$ nix-store --realise --add-root /tmp/pintest/agent --indirect $drvB
/tmp/pintest/agent
$ readlink /tmp/pintest/agent
/nix/store/…-pin-test2                    # link moved to B; A now unrooted → GC-eligible
Gotcha that bites the integration: with --add-root, nix-store --realise prints the root link path (/tmp/pintest/agent), not the store path. Today nixCopy.ts does realiseRes.stdout.trim() and feeds it straight into ${agentPath}/bin/${binary}. Two ways to handle it:
  • Resolve it — readlink -f / realpath the printed link to recover the immutable store path, and spawn against that. Cleanest: a running agent stays pinned to its exact closure even after the “latest” link moves under it on a later deploy.
  • Spawn through the link — ${linkPath}/bin/${binary} works too (the symlink resolves at exec time). Simpler, but a redeploy repoints the link, so new spawns follow the new agent while the old process keeps running. Coherent, just be deliberate about it.

3. Where the root path should live

ConstraintChoice
Whose store?The remote’s. The path being protected is on the target host, so the root must be created there — i.e. inside the ssh $host … realise, which it already is. (Localhost: same code, roots the local store. No special-case.)
Writable by?The ssh user → under $HOME. With --indirect that’s fine. e.g. ~/.local/state/drishti/gcroots/<binary>. Prefer state over cache — a GC root is durable state, and ~/.cache is exactly the thing cleaners wipe.
Keyed on what?The binary name (or the drv’s name component). One logical agent → one moving link → “latest of this agent.” Different remotes have independent stores and independent roots, so no cross-host coordination needed.
Pre-create dir?Yes — nix-store --add-root writes the symlink but not its parent dirs. Prefix with mkdir -p.

Concretely, the realise step becomes roughly

ssh $host sh -c '
  mkdir -p ~/.local/state/drishti/gcroots &&
  nix-store --realise \
    --add-root ~/.local/state/drishti/gcroots/$binary --indirect \
    $drvPath
'

Wrapping in sh -c (or resolving $HOME via a probe first) keeps the ~ expansion and the mkdir on the remote. Note this means buildSshProbeCommand needs to carry a small shell snippet rather than a bare argv — a minor shape change to the realise call.

4. What you do not need to pin

  • The .drv and its build inputs/sources. They aren’t in the output’s runtime closure, so rooting the output doesn’t hold them — and that’s correct. You don’t need the .drv after realise; a redeploy re-copies it. Rooting only the output is the right minimal pin.
  • Anything on the sender. The local .drv is re-evaluated from nix eval …drvPath each deploy; let it be GC’d.

Recommendation

  1. In surface-nix-host nixCopy.ts, add --add-root <remote $HOME state path keyed on binary> --indirect to the realise, with a preceding mkdir -p.
  2. Account for --add-root printing the link path: realpath it back to the store path for the spawn (recommended), or spawn through the link.
  3. One fixed link per agent = automatic “latest”: new deploy pins the new closure, drops the old. No pruning logic.
Net: the agent survives remote GC, reconnects stop rebuilding, and stale versions get collected on their own — all from a single flag at the seam you already have.

Note: the change is in the @kolu/surface-nix-host dependency (see /tmp/kolu-fork/packages/surface-nix-host/src/nixCopy.ts), not in this repo’s own source. Drishti consumes provisionAgent() via HostSession; the spawn path in buildAgentCommand would pick up whichever agentPath the realise returns, so adopting the fix is a dependency-side edit + version bump here.

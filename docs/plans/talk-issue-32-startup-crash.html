Issue #32 — monitor crash-loops when any initial host is unreachable

Regression from PR #31. One offline host at boot kills monitoring of all hosts and the HTTP port never binds.

Recommendation: fix the seam in kolu (Plan B), not the symptom in drishti (Plan A).

The volatility "host is unreachable over SSH" is already fully owned by HostSession. The crash happens because drishti runs the arch probe — a second, fatal SSH round-trip — before the resilient session is ever instantiated. Move the probe inside the session's spawn cycle and the crash class disappears at the root. Both reviewer lenses (Lowy/volatility, Hickey/simplicity) landed here independently.

What actually breaks

Two SSH round-trips touch the same reachability axis, handled inconsistently:

Round-tripWhereOn unreachable host
Arch probe — picks which agent .drv matches the host's nix-system archMap.ts:19 → kolu resolveSystem arch.ts:48 throws (exit 255) — and has no ConnectTimeout (host.ts:63) / no timeout in runCapture (process.ts:56), so it can also hang forever

drishti resolves the drv (probe, fatal) before creating the session (resilient):

And serve(...) (main.ts:128) runs after await buildHostRegistry (main.ts:95) — so the port never binds and launchd respawn-loops.

The asymmetry that matters

A host that drops later was probed successfully once → it has a drv → it has a session → it flows through the resilient machinery. A host unreachable at boot never gets a drv → no session is ever created → the machinery never engages. The session pool key is (host, drvPath, binary) hostSession.ts:522, so the drv is woven into session identity. The "Reconnect" button (admin-router.ts:85) calls entries.get(host)?.session.reconnect() — a silent no-op for a host that has no session.

Why not the issue's own suggestion (Plan A, drishti-only)

Issue #32 suggests Promise.allSettled + bind-first. That stops the crash, but to satisfy its own "expected behavior" (failed host stays visible and reconnectable) drishti would need to hold a sessionless "pending" entry and retry it with a new registry-level backoff loop. Both lenses flag that as the wrong move:

  • Hickey The registry backoff loop is concept multiplication — a second implementation of "recover an unreachable host" running in series with the session's existing five-state machine. The hand-off (pending → session) is an implementation accident, not a domain concept.
  • Hickey A has()-true-but-no-handler entry complects "host is in the set" with "host has a live session," forcing a new null-branch into snapshot, getHandler, the WS-upgrade guard (main.ts:202), and overloading reconnect() to mean two things.
  • Lowy Same axis, two encapsulations, now spanning two repos — any future retry-policy change has a blast radius across both. Violates the registry's own stated invariant ("no shadow data store; admin and registry cannot diverge by construction").
  • Lowy A bind-first placeholder drvPath would mint a second pooled session under a different key when the real drv resolves — a correctness hazard, not an optimization.

Plan B — fold the probe into the session spawn

One concept, one place. The probe failure becomes an ordinary provision failure flowing through the machinery that already owns this axis.

Pool-key narrowing is a correctness improvement

drvPath was in the key only as a defensive guard against a caller passing two drvs for one host — which drishti never does (one entry per host). Dropping it means a host whose nix-system changes is picked up automatically on the next reconnect, instead of stranding a stale session under a dead key. Both lenses confirmed no caller distinguishes two sessions to one host by drv.

Sequencing

  1. kolu PR: resolveDrvPath thunk + pool-key narrowing → cut release.
  2. drishti: re-pin kolu in the flake; make buildEntry synchronous; delete the Promise.all seed.
  3. Keep serve()-before-seed as cheap defense-in-depth in the same drishti PR (it's no longer load-bearing, but a future eager-await regression can't re-break the port).
  4. Update README.md if it documents startup/host-failure behavior.

Not phased: this is a bug fix, not user-visible feature work — the kolu→drishti order is natural sequencing, not a feature split. If the kolu re-pin must be deferred, the only non-regrettable stopgap is bind-first + omit-failed-hosts (drop them from the registry entirely, no parallel backoff, no sessionless entries) — degraded (failed hosts invisible until they're re-added) but not a structural fork. Do not ship Plan A's registry retry loop as an interim.

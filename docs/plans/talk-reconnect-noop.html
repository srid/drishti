“Reconnect” does nothing — diagnosis & fix

The button is a genuine silent no-op for nix copy-driven failures. Root cause is in the @kolu/surface-nix-host library, not drishti. Three deliverables below: the bug fix, the tip cleanup, and the .agency/code-policy.md rule.

BUG  Why nothing happens — and no console error

The click reaches the server fine. The dead end is inside the library’s HostSession.reconnect().

1. Button → onReconnect calls the RPC; .catch only logs. App.tsx:616–622, 1060–1066
2. Admin router: has() check passes, then registry.reconnect(host), then return { ok: true } — unconditionally. admin-router.ts:85–92
3. registry.reconnect → session.reconnect() (returns void). hostRegistry.ts:171–173
4. HostSession.reconnect() hits its guard and silently returns. kolu hostSession.ts:415–423

The guard that fires:

reconnect(): void {
  if (this.destroyed || this.refCount === 0) return;
  if (this.clientPromise !== null || this.pendingTimer !== null) return;  // ← returns here
  ...
}
// doc comment claims: "The give-up gate left pendingTimer and
// clientPromise null, so a genuinely-failed session always passes." ← FALSE for this path

clientPromise is nulled in only two places: handleChildDone (the ssh child’s exit/error handler) and teardown. kolu hostSession.ts:350–356, 460–472

But the screenshot’s failure is 'nix copy --derivation' exited with code 1 — a provision failure. spawn() throws on that branch before any ssh child is spawned, so handleChildDone never runs and clientPromise keeps the last rejected promise. kolu hostSession.ts:311–325; nixCopy.ts:76–81

The give-up branch then sets connection:"failed" and returns without nulling clientPromise. kolu hostSession.ts:436–444

So in the terminal failed state reached via nix copy: clientPromise !== null → reconnect() early-returns → no spawn, no state change, no error. The RPC still resolved { ok: true }, so the client .catch never fires. That is the “not even errors” symptom.

Confirmed against both the pinned revision (npins → juspay/kolu@8a2bc6d) and current kolu master — the bug is live in both. A failure driven by agent exit (not provision) does null clientPromise, so Reconnect would work there; this is specific to the nix copy / nix-store --realise provision paths.

FIX A  One line in kolu — make failed honor its invariant

The terminal failed state’s documented invariant is “no spawn in flight.” Make it true. In the give-up branch of scheduleReconnect:

kolu/packages/surface-nix-host/src/hostSession.ts  (~line 443)

  if (this.consecutiveFailures >= MAX_CONSECUTIVE_FAILURES) {
    this.addLocalProgress(`gave up after ...`);
+   this.clearClientPromise();           // failed ⇒ nothing in flight
    this.updateState({ connection: "failed" });
    return;
  }

This makes reconnect()’s existing guard correct as already documented — no guard change needed.

Rejected the alternative. Keying the guard off connection === "failed" instead would treat the symptom, leave the false doc comment standing, and pass the guard even if a spawn were genuinely in flight. Both reviewers said fix the invariant, not the guard.

Verified safe: nulling clientPromise does not break the bridge. waitForNextClient treats currentClient() === null as “keep waiting,” then wakes on the next state change — which reconnect()→spawn()→updateState("copying") delivers. kolu waitForNextClient.ts:36–48

Hardening (folded in from Hickey)

clientPromise = null would now live in three non-adjacent sites, the “null it before failed/teardown” rule enforced only by memory. Extract a named private clearClientPromise() and route all three through it, so the invariant is searchable rather than conventional. Update the now-true reconnect() doc comment to name the give-up branch as its enforcer. kolu hostSession.ts:354, 471, 443; comment 413

Boundary check (Lowy): connection-lifecycle volatility is fully encapsulated in kolu; drishti only mirrors state and calls reconnect(). The fix correctly lands in kolu — a drishti-side workaround (destroy+recreate, poking internals) would leak the state machine across the repo boundary. Ship kolu, then npins-bump drishti.

FIX B  The tip is a guess — show the real error instead

You’re right that the 💡 tip fires regardless of cause. Worse: it duplicates text the session already produced. The exact “trusted-users” line is emitted into progressLines at give-up kolu hostSession.ts:438, and the real nix copy stderr is already streamed there too — but FailedCard shows none of it. It shows only the terse lastError (“exited with code 1”) plus a hardcoded speculative <div>. App.tsx:1048–1059; surface.ts:154–162

Minimal fix: pass c().progressLines into FailedCard, render the tail (the actual failure output), and delete the hardcoded tip. The real cause replaces the guess — no string-matching, no new coupling.

FIX  Reconnect outcome — no return contract needed

Tempting to make reconnect() / the RPC report “did it re-arm?”. Don’t. The connection cell already streams the real outcome (copying → connecting → connected, or back to failed) as the single source of truth the overlay reads. A return value would duplicate a cell transition and invite a two-sources race. Once Fix A lands, clicking Reconnect visibly drives the card through copying… — that is the user-visible feedback. router.ts:183–189; App.tsx:1029

POLICY  .agency/code-policy.md — errors must be visible

New file. Draft:

# Code Policy

## Errors must reach the user, never the void

A failure the user can act on must surface in the UI (or logs the user
reads). Swallowing it — or replacing it with a guess — is a bug.

### Banned
- `.catch(() => {})` / empty `catch {}` that drops an actionable error.
  Logging to the browser console is **not** user-visible; the console is
  a developer surface, not a user one.
- Fire-and-forget calls whose only failure mode is a silent no-op, with
  the caller (or RPC) reporting success regardless. If an action can
  fail to take effect, its effect — or its failure — must be observable.
  (See: the "Reconnect" no-op — `reconnect()` returned void, the RPC
  returned `{ ok: true }`, and a dead session looked re-armed.)
- Speculative "tips" / remediation text shown unconditionally, decoupled
  from the actual error. A tip that may not apply is noise that buries
  the real cause. Show the captured error; gate any hint on a structured
  signal that actually identifies the cause.

### Required
- Prefer surfacing the real captured failure output over a hand-written
  guess about what went wrong.
- When state is streamed (e.g. a connection/status cell), let the stream
  be the single source of truth for success/failure — don't bolt on a
  parallel return value that can disagree with it.
- If you must catch-and-continue for control flow, the error still has to
  land somewhere a human will see it (state cell, surfaced log, toast).

Workflow rule reminder: this is user-facing, so README.md gets a touch if the Reconnect behavior is described there (it is — the “failed state … Reconnect button” paragraph).
